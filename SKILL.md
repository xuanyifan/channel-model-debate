# 渠道模型交叉辩论 (Channel Model Debate)

多模型 AI 渠道交叉辩论，迭代优化直至共识最佳实践。

## 两种使用模式

### 模式A：设计方案辩论（头脑风暴）

从零开始设计新方案，多模型交叉论证迭代优化。

**典型场景**：
- 设计用户认证系统方案
- 讨论微服务拆分策略
- 制定技术选型决策

### 模式B：代码实现交叉验证

已有设计/计划文档，对照文档验证现有代码实现是否合规，发现不一致之处。

**典型场景**：
- 对照设计文档检查接口实现是否合规
- 验证业务逻辑是否符合计划文档规定
- 检查异常处理是否覆盖文档描述的场景

## 工作流程

### 0. 判断模式

仔细理解用户的议题，判断使用哪种模式：

| 关键词 | 模式 |
|--------|------|
| 设计、方案、头脑风暴、规划 | 模式A |
| 验证、是否符合、对照检查、回归 | 模式B |

如果不确定，根据上下文判断：有时两种模式会结合使用（先设计后验证）。

### 1. 根据用户描述查找文件

用户会提及所参考的文档/代码，Skill 根据描述在当前目录自动查找匹配文件：

1. **用户明确提到的文件** — 用户在议题中具体指定的文件
2. **用户描述的文档** — 如"接口文档"、"设计文档"等描述，自动在当前目录扫描匹配：
   - `*.md`（设计文档、接口文档、README）
   - `*.yaml` / `*.yml`（配置文件、OpenAPI spec）
3. **相关代码** — 如涉及具体模块代码，自动查找：
   - `*.py` / `*.java` / `*.ts`（限制单文件2000字符）
4. **跳过** — `node_modules`、`__pycache__`、`.git`、`target`、`dist`、`build`

**扫描策略**：
- 用户提到"XXX文档" → 在当前目录模糊匹配含"XXX"的文件
- 用户提到"XXX接口" → 匹配含"XXX"的接口定义文件
- 用户提到"XXX模块" → 匹配含"XXX"的代码文件

**扫描深度**：最多遍历 3 层目录，上限 30KB 文本总量。

### 1.5 上下文压缩（动态取限 + 一次性压缩）

Skill 在将上下文传给脚本前，根据 config.json 中各渠道的上下文限制决定是否压缩并写入单一 `context`。脚本无需感知，零改动。

1. **读取原始内容**：将从步骤1找到的文件内容汇总，计算总字符数
2. **获取各渠道上下文限制**：从 config.json 各 channel 的 `context_limit_tokens` 字段读取 token 上限。若字段缺失、为 null 或 0，视为无限制。按 ~4 chars/token 将 token 限制换算为字符上限
3. **确定压缩目标**：取所有渠道 `context_limit_tokens` 中的最小值，按 ~4 chars/token 换算为字符上限，作为压缩目标。若所有渠道均无限制，不压缩
4. **决策**：
   - 总字符数 ≤ 最短限制 → 直接传递原始内容，不压缩
   - 总字符数 > 最短限制 → 做语义摘要压缩，目标长度 = 最短限制的 ~80%（保留 margin 给 prompt 模板和辩论历史），保留接口定义、业务逻辑、错误码等关键信息
5. **输出格式**：写入 `/tmp/debate_context.txt`，JSON 格式供脚本 `--ctx-file` 读取：

```json
{
  "context": "压缩后上下文（超限模型使用）...",
  "context_full": "原始完整上下文（未超限模型使用，无需压缩时等同于 context）...",
  "over_limit": ["MiniMax-M2.7-highspeed", "GLM-5.1"],
  "window": {
    "max_chars": 30000,
    "min_preserve_rounds": 2
  }
}
```

- `context`：压缩后的文本，由脚本按 `over_limit` 列表分发给超限模型
- `context_full`：原始完整上下文，分发给未超限模型。若所有模型均未超限，此字段可省略
- `over_limit`：超限的 model 名列表（与 config.json 中 `model` 字段值完全一致，大小写敏感）。脚本据此判断：model 在此列表 → 用 `context`，否则 → 用 `context_full`
- `window.max_chars`：辩论历史累计文本的字符上限，超限触发滑动窗口
- `window.min_preserve_rounds`：滑动窗口最少保留的最近轮次数

模型升级或新增渠道时，手动更新 `config.json` 中对应 `context_limit_tokens` 即可。

### 2. 调用辩论脚本

**命令**：
```bash
# 模式A（设计/方案）
script -q /dev/null python3 /Users/huskar/Documents/Git/channel-model-debate/channel_debate.py "议题" --search --ctx-file /tmp/debate_context.txt > /tmp/debate_output.txt 2>&1 &

# 模式B（验证/检查）
script -q /dev/null python3 /Users/huskar/Documents/Git/channel-model-debate/channel_debate.py "议题" --ctx-file /tmp/debate_context.txt > /tmp/debate_output.txt 2>&1 &

# 回归验证不通过 → 追加修复
script -q /dev/null python3 /Users/huskar/Documents/Git/channel-model-debate/channel_debate.py "议题" --resume /tmp/debate_state.json > /tmp/debate_output.txt 2>&1 &
```

> `--search` 是可选的联网搜索，模型辩论中对不确定的内容自行发起，不区分模式。
- `script -q /dev/null`：创建伪终端，强制行缓冲，确保实时写入
- 输出重定向到 `/tmp/debate_output.txt`，Skill 每 3 秒增量读取
- 后台运行（`&`），不阻塞
- 共识后自动保存状态到 `/tmp/debate_state.json`

### 3. 执行辩论（流式输出 + 阶段概览）

**⚠️ 禁止跳过：不能以"代码审查"、"手动分析"替代本步骤。必须启动多渠道辩论脚本。**

#### 3.1 启动辩论

使用后台模式运行，输出重定向到临时文件，使用 `script` 伪终端强制行缓冲：

```bash
script -q /dev/null python3 /Users/huskar/Documents/Git/channel-model-debate/channel_debate.py "【议题】" --ctx-file /tmp/debate_context.txt > /tmp/debate_output.txt 2>&1 &
echo $!  # 记录 PID
```

#### 3.2 建立阶段任务看板

**🚫 3.2 不可跳过：3.1 启动辩论脚本后，必须立即执行 3.2 建立任务看板，然后才能进入 3.3 轮询。禁止 3.1→3.3 跳步。**

**⚠️ 步骤0：清理残留** — 先用 `TaskList` 列出所有任务，逐一用 `TaskUpdate` 标 `deleted`，确保看板从零开始。只清理属于本技能的阶段性任务，不碰用户其他任务。

**步骤1：逐一创建** — 严格按下列顺序逐次调用 `TaskCreate`。每次只发一个，等返回后再发下一个，严禁在同一条消息中并行：

| 序号 | 任务名（用作后续引用标识） | 说明 | 归属 |
|------|--------------------------|------|------|
| 1 | 第1轮：初始方案生成 | 渠道并发独立论点 | 脚本 |
| 2 | 第2-N轮：交叉辩论 | 互相挑刺迭代优化，每轮后共识检测 | 脚本 |
| 3 | 共识达成 & 最终汇总 | 收敛后调用 synthesize 输出草稿 | 脚本 |
| 4 | 回归验证 | 逐项检查共识结论可行性，不通过则回辩论修复 | Skill |
| 5 | 交付 | 验证通过后综合输出结论 | Skill |

**步骤2：状态转移** — 创建后立即将任务1标 `in_progress`。每个任务完成后先标 `completed`，再激活下一个。当前进行中最多 1 个。任务1-3的状态由脚本输出流中的标记判定（见 3.3），任务4-5由 Skill 手动判定。

#### 3.3 流式轮询

**🚫 硬性禁令：绝对禁止以下偷懒行为：**
- ❌ `sleep 30` 或 `sleep 60` 等固定长等待后一次性读取
- ❌ `while kill -0 $PID; do sleep 5; done` 死等不输出
- ❌ 把轮询交给后台任务（`run_in_background`）让用户看不到进度
- ❌ 等待脚本结束后一次性 `cat` 全部输出

**✅ 必须：增量读取 + 即时反馈。每个标记出现时立即在对话中报告，用户必须实时看到辩论进度。**

**⚠️ 每 3 秒读取一次 `/tmp/debate_output.txt`，有新内容立即输出，无新内容则继续等：**

```
while true; do
  # 1. 增量读取：只读上次之后的新增行
  new=$(tail -n +$((last_line+1)) /tmp/debate_output.txt 2>/dev/null)
  [ -n "$new" ] && echo "$new" && last_line=$(wc -l < /tmp/debate_output.txt)

  # 2. 根据标记立即反应
  新内容有 "⏳" → 报告进度：正在等待哪个模型
  新内容有 "✅" → 报告完成：展示模型论点摘要（前200字）
  新内容有 "🚫" → 报告死渠道：标注该渠道已跳过
  新内容有 "📊" → 报告对比：展示模型间共识/分歧
  新内容有 "🔍 检查是否达成共识" → 等待共识判定
  新内容有 "【共识结论草稿】" → **不得直接交付！** 必须执行第4步回归验证
  新内容有 "💾" → 脚本已保存状态，正常结束
  新内容有 "❌ 未找到 config.json" → 缺少配置文件，立即报告用户

  # 3. 终止检查
  进程不在且读到 "💾" → 退出循环，进入步骤4
  进程不在且未读 "💾" → 报告进程异常退出的当前阶段

  # 4. 仅在没有新内容且进程存活时短暂休眠
  [ -z "$new" ] && sleep 3
done
```

**展示规则：每个标记出现时必须立即在对话中反馈，不得批量累积后一次性报告。**

**⚠️ 关键规则：synthesize 输出是草稿，不是最终交付件。必须经过第4步回归验证。**

#### 3.4 进程生命周期

- 脚本正常结束且读到"💾" → 进入步骤4回归验证
- 脚本被 kill → 报告当前阶段（_CURRENT_PHASE + _CURRENT_ROUND），询问是否重试
- 脚本异常退出且无 "💾" → 报告失败，展示最后输出

### 4. 回归验证（交付前必做，不可跳过）

#### 4.1 验证清单

**模式A（设计辩论）**：

| 验证项 | 检查方法 |
|--------|---------|
| 1. 技术栈兼容性 | 对照当前项目依赖和技术选型，检查共识结论是否有冲突 |
| 2. 边界场景覆盖 | 穷举异常输入、极值，空值、并发、超时等场景是否被覆盖 |
| 3. 最佳论点TOP3可行性 | 逐条验证在当前项目上下文中是否可落地 |
| 4. 改进项具体方案 | 检查每个"需改进处"是否有可执行的改进方案，不能只有方向性描述 |
| 5. 结论间矛盾检查 | 交叉对比所有共识结论，检查是否存在逻辑矛盾 |
| 6. 参考文档误读检查 | 对比原始文档内容 vs 辩论中对文档的引用，确认无理解偏差 |

**模式B（代码交叉验证）**：

| 验证项 | 检查方法 |
|--------|---------|
| 1. 接口定义一致性 | 对照文档检查字段名、类型、必填/可选、默认值是否一致 |
| 2. 业务流程合规性 | 对照文档流程图逐步验证代码分支路径 |
| 3. 异常处理覆盖度 | 文档描述的所有异常场景是否都有对应处理 |
| 4. 隐藏实现 | 发现代码中存在但文档未提及的逻辑，标注并评估风险 |
| 5. 版本一致性 | 检查文档版本号 vs 代码版本号，确认是否为最新版本 |

#### 4.2 验证不通过处理

按问题严重程度分级处理：

```
┌─────────────────────────────────────────────────────────────┐
│ 严重程度  │ 判定标准               │ 处理方式                │
├───────────┼───────────────────────┼─────────────────────────┤
│ 🔴 严重   │ 结论矛盾 / 技术栈冲突  │ --resume 追加2轮修复      │
│           │ 接口定义不一致         │ → 再次回归验证           │
│           │ 安全性问题            │ → 最多重试2次            │
│           │                       │ → 2次后仍未通过则标记     │
│           │                       │   [ASK USER] 交用户决策  │
├───────────┼───────────────────────┼─────────────────────────┤
│ 🟡 中等   │ 边界遗漏 / 文档误读   │ 追加1轮"补丁辩论"        │
│           │ 需改进项无具体方案    │ 各模型针对不通过项给出    │
│           │                       │ 具体修复建议              │
│           │                       │ → 综合后直接输出修正稿    │
├───────────┼───────────────────────┼─────────────────────────┤
│ 🟢 轻微   │ 描述瑕疵 / 格式问题   │ 直接修正后标记"[已修正]"  │
│           │                       │ 附修正说明               │
└─────────────────────────────────────────────────────────────┘
```

**核心原则**：
- 验证不通过 → 进修复循环 → 通过再交付
- 严重问题最多重试2次，避免无限循环
- 2次后仍不通过，标记 [ASK USER] 交由人工决策
- 修复后必须重新跑一遍完整验证清单

**看板处理**（与 3.2 任务管理规则兼容）：
- 🟢 轻微：不操作看板，任务"回归验证"→`completed`，随后激活任务"交付"→`in_progress`→`completed`
- 🟡 中等 / 🔴 严重：先按任务名找到当前看板中的"回归验证"和"交付"，分别 `TaskUpdate` 为 `deleted`，然后执行补丁辩论 / `--resume`。修复完成后按 3.2 步骤1的规则重新创建"回归验证"→"交付"（每次只发一个 TaskCreate）

### 5. 交付（回归验证通过后执行，不可跳过）

回归验证通过后，**立即**输出以下结构化的最佳实践结论：

```
## 最佳实践结论

### 议题回顾
<原始议题>

### 多方共识
<各渠道达成一致的核心结论，按维度展开，每项有论据支撑>

### 风险与改进
| 风险/问题 | 严重度 | 改进建议 | 优先级 |
|------|:----:|------|:----:|
| ... | ... | ... | ... |

### 行动建议
<可执行的具体行动项，不是方向性描述>

### [ASK USER]（如有）
<需用户决策的问题>
```

**要求：**
- 结论必须有论据支撑，不能只给判定
- 改进建议必须可执行，不能是"建议优化"这种空话
- 不是评卷，是交付最佳实践方案

**⚠️ 此步骤不可跳过。验证通过 = 立即交付。**

## 约束

- **不过度设计**：只辩论当前议题范围内的事项，不做范围外的延伸
- **务实迭代**：达到共识或最大轮次即停止，不无限循环
- **失败容错**：单个渠道失败不影响整体，自动跳过
- **上下文压缩**：由 Skill 在启动脚本前完成，脚本不再感知 `context_limit`
- **按描述匹配**：用户提到什么就在项目里找什么，不猜测无关文件
